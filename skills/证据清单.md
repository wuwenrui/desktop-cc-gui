## 功能说明

本 Skill 用于处理**民事诉讼证据清单（证据目录）编制**事务。基于案件目录内的真实材料，生成提交法院使用的规范证据清单，输出 .docx 表格文书。

**上游输入**：优先复用「卷宗梳理」Skill 产出的《卷宗梳理工作底稿》中的证据矩阵；同时与「民事起诉状」「民事答辩状」Skill 联动——清单中的证据编号、名称、证明目的必须与诉讼文书中引用的证据保持一致。

**定位**：证据清单是法官认识案件证据体系的索引。一份好的清单让法官按图索骥，每条证明目的都指向一项诉讼请求或一个争议焦点；一份差的清单只是文件名的堆砌。

---

## 一、核心纪律（最高优先级）

1. **只列真实存在的材料**：清单中每一项证据必须对应案件目录中实际存在的文件；禁止列入"应该有但还没拿到"的材料——那属于待补证据清单（见 §七），与正式清单分开。
2. **证明目的必须具体**：一句话点明"证明……的事实"，并与诉讼请求或争议焦点呼应；禁止"证明案件事实""证明被告违约"这类空泛表述。
3. **名称、日期与原件一致**：证据名称中的文件名、日期以材料原貌为准，不得"美化"或改写；日期不明的按 §五 的降级规则处理。
4. **页码真实可核**：页码按提交版证据册的实际编排填写；尚未编册的，先输出占位符并在备注说明，定稿前回填。
5. **输出落盘**：写入案件目录「文书/」子目录（不存在则创建）。
6. **语体**：正式法律文书语体，禁用 emoji。

## 二、核心工作流

### Step 0：读取上游材料

- 读取《卷宗梳理工作底稿》的证据矩阵（如有），按 §六 的映射规则转化
- 无底稿的，遍历案件目录材料，对每份文件记录：文件名、形成日期、持有来源、与待证事实的关系

### Step 1：确定证明体系框架

- 读取起诉状（或答辩状）草稿，列出全部诉讼请求与争议焦点
- 为每项请求/焦点列出需要证明的要件事实——这是分组和编排证明目的的骨架

### Step 2：证据分组（按证明对象）

按 §四 的五大组归类；同一证据能证明多个事实的，归入最核心的组，证明目的中并列写明。

### Step 3：组内排序与命名

- 组内按证据形成时间升序排列
- 按 §五 规范统一命名，全局连续编号（跨组不重新起号）

### Step 4：撰写证明目的

按 §五.3 的写法逐条撰写，写完后做"反向核查"：每项诉讼请求/争议焦点至少有一条证据的证明目的指向它；指向不了的请求即举证短板，记入待补证据提示。

### Step 5：编排页码与备注

- 按"分组顺序+组内顺序"虚拟编排证据册页码（每份证据起始页~结束页）
- 备注栏标注：原件/复印件、原件持有人、电子数据原始载体情况等质证关注点

### Step 6：生成 .docx 输出

- 按 §八 格式生成表格文书，预留质证意见空列
- 文件命名：`证据清单_【提交方简称】_YYYYMMDD.docx`
- 同时输出待补证据提示（如有，作为独立内部区块，提交前删除）

## 三、清单列结构（固定，不得增删核心列）

| 列名 | 填写规范 |
|------|----------|
| 序号 | 全局连续编号：1、2、3…… |
| 证据名称 | 按 §五 命名规范，含形成日期与文件名 |
| 证据来源 | 谁持有/从哪取得：原告留存原件、被告处取得、银行打印、公示系统下载、微信导出等 |
| 证明目的 | 一句话"证明……的事实"，呼应请求/焦点，见 §五.3 |
| 页码 | 证据册起止页，如"第1-5页" |
| 备注 | 原件/复印件、份数、电子数据载体、需当庭出示原件等提示 |
| 质证意见 | **预留空列**，表头保留、内容留空，供庭审记录对方质证意见 |

## 四、分组方法论（按证明对象五大组）

| 组别 | 收纳什么 | 典型证据 |
|------|----------|----------|
| 一、主体资格类 | 证明双方当事人身份、资格、关联关系 | 营业执照/登记信息、身份证复印件、法定代表人身份证明、授权委托书相关 |
| 二、法律关系成立类 | 证明合同/侵权等法律关系的成立与内容 | 合同及补充协议、订单、借条、章程、框架协议 |
| 三、履行情况类 | 证明各方履行与违约的过程事实 | 送货单、验收单、转账回单、对账单、催告函、聊天记录 |
| 四、损失计算类 | 证明损失金额、利息违约金计算基础 | 发票、结算单、评估/鉴定报告、维修票据、利率依据材料 |
| 五、其他 | 程序性及辅助性材料 | 时效中断证据、公证书、保全相关、行政处理记录 |

分组纪律：
- 分组的本质是**按"要证明什么"组织，而不是按文件类型组织**——同是微信记录，证明合同成立的入第二组，证明催款的入第三组
- 组标题行在表格中以合并单元格呈现，如"第一组：主体资格类证据"
- 案情简单（证据少于 8 份）时可不分组，但组的逻辑仍体现在排序上
- 答辩方使用本 Skill 时，分组对应调整为反驳对象（如"反驳原告履行主张类"），方法不变

## 五、命名与撰写规范

### 5.1 证据命名规范

```
证据X：YYYY年MM月DD日《文件名》
```

示例：

```
证据3：2024年3月12日《工业品买卖合同》
证据7：2024年6月20日银行转账回单（金额500,000元）
证据9：2025年1月10日微信聊天记录（原告业务员张某与被告采购王某）
```

命名细则：
- 有正式标题的书证用书名号引原标题；无标题的（回单、照片、聊天记录）用"类型+关键识别信息"描述
- 日期取**证据形成日期**（合同取落款日、回单取交易日、聊天记录取起止日）；仅能确定年月的写"YYYY年MM月"；完全不明的省略日期并在备注注明"形成日期不详"
- 多份同类证据可合并为一项（如"银行转账回单3张"），但金额、日期在证明目的或备注中分别列明

### 5.2 证据来源写法

写清楚"从哪来、谁持有"，这直接关系证据合法性与真实性质证：

```
原告留存原件 / 原告自被告处签收取得 / 原告开户银行柜台打印并加盖业务章 /
国家企业信用信息公示系统下载打印 / 原告手机微信聊天记录导出（原始载体为
原告业务员张某手机，可当庭出示）
```

### 5.3 证明目的写法（一句话规范）

固定句式：**"证明……的事实"**，且必须满足两个条件：
1. 指向具体事实（有主语、有行为、有时间或金额要素）
2. 与某项诉讼请求或争议焦点呼应

正反对照：

| 不合格写法 | 合格写法 |
|------------|----------|
| 证明案件基本事实 | 证明原、被告于2024年3月12日成立买卖合同关系及货款总额、付款期限、违约金标准的约定（对应诉讼请求一、二） |
| 证明被告违约 | 证明付款期限届满后被告仅支付500,000元、尚欠486,000元的事实（对应诉讼请求一） |
| 证明原告催款 | 证明原告于2025年1月10日向被告催告付款、诉讼时效中断的事实 |

### 5.4 页码编排规则

- 证据册全册连续编页，每项证据填起止页："第12-18页"
- 未实际编册前，页码列填"【编册后回填】"，并在文末提示装册顺序即清单顺序
- 单页证据写单页页码，不写区间

## 六、与卷宗梳理证据矩阵的衔接（字段映射）

已有《卷宗梳理工作底稿》的，证据矩阵按下表直接转化，避免重复劳动：

| 证据矩阵字段（卷宗梳理） | 证据清单字段（本 Skill） | 转化动作 |
|--------------------------|--------------------------|----------|
| 证据名称 | 证据名称 | 按 §5.1 重新规范命名（补形成日期） |
| 来源文件 | 证据来源 | 由"文件路径"改写为"持有/取得方式"表述 |
| 能证明什么 | 证明目的 | 升格为"证明……的事实"句式，补充与请求/焦点的呼应 |
| 弱点/瑕疵 | 备注 | 转化为质证应对提示（如"复印件，原件在被告处，将申请法院责令提交"） |
| —— | 页码 | 新增，按编册顺序编排 |
| —— | 质证意见 | 新增空列 |

转化纪律：矩阵中标注「待核实」的证据，核实前不进入正式清单，进入 §七 的待补证据提示。

## 七、待补证据提示（内部区块）

正式清单之后附独立区块（提交法院前删除）：

```
——以下为内部工作内容，提交法院前删除——

待补证据提示
1. 诉讼请求二（违约金）目前仅有合同条款支持起算标准，缺少证明货物验收日期
   的证据（影响起算日认定），建议补充验收单或以催告函送达日为替代起算点。
2. 证据9微信聊天记录需在开庭时出示原始载体（张某手机），请提前安排。
3. 对账单（2025.3.31）金额与合同总价不一致，未列入本清单，待委托人说明差异
   原因后决定是否补充提交。
```

## 八、文书格式规范（.docx）

使用 python-docx 生成：

| 要素 | 规范 |
|------|------|
| 纸张 | A4 横向（列多，横向排版可读性好）；页边距上下 2.54cm、左右 2.5cm |
| 标题 | "证据清单"或"证据目录"，二号宋体加粗居中；下一行小字注明案件名称与提交方 |
| 表头 | 宋体小四加粗，居中，浅灰底纹 |
| 表格正文 | 宋体五号（10.5pt），行距单倍，单元格上下留 0.1cm 边距 |
| 列宽建议 | 序号5%、证据名称22%、证据来源15%、证明目的30%、页码8%、备注10%、质证意见10% |
| 组标题行 | 合并单元格通栏，宋体小四加粗，如"第一组：主体资格类证据" |
| 落款 | 表格下方右对齐："提交人：（留空）　　年　月　日（留空）" |

## 十、操作 Checklist

**编制前：**
- [ ] 卷宗梳理证据矩阵已读取（或已遍历材料自建台账）
- [ ] 诉讼请求与争议焦点清单在手
- [ ] 「待核实」证据已剔出，转入待补证据提示

**内容质量：**
- [ ] 每项证据在案件目录中真实存在
- [ ] 命名符合「证据X：YYYY年MM月DD日《文件名》」规范
- [ ] 证明目的均为"证明……的事实"句式且指向具体请求/焦点
- [ ] 反向核查完成：每项请求至少有一条证据指向，短板已记入待补提示
- [ ] 分组归属按证明对象而非文件类型
- [ ] 备注栏已标注原件/复印件与载体情况

**输出：**
- [ ] 质证意见空列已预留
- [ ] 页码已编排或已用占位符+回填提示
- [ ] .docx 格式符合 §八 规范，落款留空
- [ ] 待补证据提示已附且标注"提交前删除"
- [ ] 文件已落盘至「文书/」子目录

## 十一、注意事项

1. **清单与文书联动一致**：起诉状/答辩状中引用的证据编号、名称必须与本清单完全一致；任何一方修改后，另一方同步更新。
2. **不利证据的处理是律师决策**：本 Skill 对委托人不利的材料默认不列入提交版清单，但必须在待补证据提示中向律师报告该材料的存在及风险，由律师决定是否提交——隐瞒不报是不合格输出。
3. **复印件风险显性化**：仅有复印件的关键证据，备注中写明原件下落及补强方案（申请调取、申请责令对方提交等），不能只写"复印件"三个字。
4. **电子数据三件套**：聊天记录、电子邮件类证据，备注中固定核查三件事——原始载体是否可出示、导出过程是否可复现、主体身份能否对应（微信号与当事人的关联证明）。
5. **份数与装订**：提交版按"正本一份+副本按对方当事人人数"准备；清单顺序即装册顺序，编页后回填页码列。
6. **法条引用纪律**：涉及证据规则的（如《最高人民法院关于民事诉讼证据的若干规定》），写法律/司法解释名称+条款号，并提示"以现行有效版本为准，引用前核对"。
