## 功能说明

本 Skill 用于处理**诉讼案件卷宗梳理**事务。用户将案件材料目录作为上下文提供后，本 Skill 系统化通读全部材料，产出一份结构化的「卷宗梳理工作底稿」，作为后续起草起诉状、答辩状、证据清单、代理词等一切诉讼文书的事实基础。

**定位**：卷宗梳理是诉讼工作的第一步，也是其他文书类 Skill（民事起诉状、证据清单、民事答辩状）的上游输入。底稿质量直接决定后续文书质量。

---

## 一、核心纪律（最高优先级，贯穿全程）

以下纪律优先级高于一切输出效率考虑，违反任何一条即视为底稿不合格：

1. **事实必须带出处**：底稿中的每一条事实，必须标注来源文件名 + 定位（页码、段落或条款号）。无法定位来源的"事实"不得写入事实部分。
2. **禁止编造或推测**：禁止输出任何未见于材料的事实；禁止用"通常""应该是"等推测填补材料空白。法律分析可以推理，事实记载不可以。
3. **不确定标注「待核实」**：材料字迹不清、扫描缺页、日期残缺、金额模糊的，照实记录可辨认部分，并标注「待核实」及核实建议（如"建议向委托人调取原件"）。
4. **区分「材料记载」与「当事人陈述」**：委托人口头说明、微信聊天中的单方说法，与合同书证的证明力不同。底稿中用"据《XX》记载"与"据委托人陈述/据微信记录中XX所述"区分表述。
5. **中立梳理，不预设立场**：卷宗梳理阶段对双方有利不利的事实一律记录，不因代理立场过滤对己方不利的材料——不利事实恰恰是庭前必须知道的。
6. **输出落盘位置**：全部产出物写入案件目录下的「文书/」子目录（不存在则先创建），不污染原始材料目录，不修改任何原始文件。

## 二、通读方法论

### 2.1 通读顺序（必须按此顺序，不得跳跃）

| 轮次 | 材料类型 | 阅读目的 |
|------|----------|----------|
| 第一轮 | 诉讼文书（起诉状/答辩状/判决书/裁定书/受理通知/传票/保全裁定等） | 最快建立案件框架：谁告谁、告什么、程序走到哪一步 |
| 第二轮 | 合同及交易文件（合同、补充协议、订单、对账单、结算单、收据、发票、转账凭证） | 确定法律关系的骨架：权利义务、金额、期限、违约条款、管辖条款 |
| 第三轮 | 沟通记录（微信/短信/邮件/通话录音整理稿/会议纪要/函件） | 还原履行过程和争议发生过程，寻找自认、催告、变更合意等关键节点 |
| 第四轮 | 其他材料（主体资料、鉴定报告、照片、现场记录、公证书、委托人自述等） | 补全主体信息、损失证据和背景事实 |

**为什么是这个顺序**：诉讼文书是双方对案件的"摘要"，先读可避免在海量原始材料中迷失；合同决定法律关系定性，是衡量后续一切履行行为的标尺；沟通记录数量大、噪音多，必须带着前两轮形成的问题清单去读，否则效率极低。

### 2.2 边读边记（每轮通读的随手产出）

每读一份文件，立即记录四元组：

```
文件名 | 文件性质（书证/电子数据/陈述） | 关键内容摘要（≤3行） | 提取出的事实节点（日期+事件）
```

第三轮读沟通记录时额外记录：对方在沟通中**承认过什么**（潜在自认）、己方**承诺过什么**（潜在风险）。

### 2.3 文件清点规则

- Step 0 必须先输出文件清单（含子目录），逐一标注"已读/无法读取/疑似缺页"
- 无法读取的文件（加密、损坏、纯图片无文字且无法辨认）列入缺料清单，不得跳过不提
- 同一文件存在多个版本（如合同扫描件与照片版）的，比对是否一致，不一致即记入矛盾点清单

## 三、核心工作流

### Step 0：进入案件目录，清点文件

- 列出目录内全部文件（含子目录），输出文件清单
- 识别文件类型，按 §2.1 的四轮顺序排出阅读计划
- 确认「文书/」子目录是否存在，不存在则创建

### Step 1：第一轮——诉讼文书

- 提取：案号（如有）、法院、程序阶段（诉前/一审/二审/执行）、双方当事人及诉讼地位、诉讼请求、对方主张的事实版本
- 产出：案件基本信息表的初稿 + 程序时间节点（举证期限、开庭日期，如材料中有）

### Step 2：第二轮——合同及交易文件

- 提取：法律关系类型、合同主体（精确到一字不差）、标的与金额（大小写均记）、履行期限、付款节点、违约条款（条款号+内容）、解除条款、管辖/仲裁条款
- 产出：法律关系定性初判 + 事实时间轴的骨架节点

### Step 3：第三轮——沟通记录

- 带着问题读：付款/交付实际发生了没有？何时发生？双方对违约的态度？有无催告、对账、还款承诺、变更约定？
- 产出：事实时间轴的血肉节点 + 自认/不利陈述摘录（注明哪一方、什么时间、原文摘录）

### Step 4：第四轮——其他材料

- 提取：当事人主体信息（自然人身份信息/法人登记信息）、损失证据、鉴定结论、现场情况
- 产出：补全基本信息表 + 证据矩阵中的辅助证据条目

### Step 5：交叉比对——找矛盾

- 比对维度：日期矛盾（合同落款 vs 转账时间 vs 聊天记录时间线）、金额矛盾（合同金额 vs 发票 vs 对账单 vs 实际转账）、主体矛盾（签约方 vs 实际收款方 vs 实际履行方）、版本矛盾（多版本合同条款差异）
- 每个矛盾点记录：矛盾内容 + 涉及的两份（或多份）文件名与定位 + 对案件的潜在影响

### Step 6：缺料检查

- 对照 §五 的按案由举证要求表，逐项核对材料中是否具备
- 缺失项写入缺料清单，注明：缺什么、为什么需要（对应哪个待证事实）、建议向谁调取

### Step 7：输出工作底稿

- 按 §四 的六模块结构组装，输出 Markdown 文件至「文书/」目录
- 文件命名：`卷宗梳理工作底稿_【案件简称】_YYYYMMDD.md`
- 用户要求 docx 的，用 python-docx 另存一份（标题黑体加粗、正文宋体 12pt、表格带边框）

## 四、工作底稿结构与输出模板（六大模块）

### 4.1 模块一：案件基本信息表

| 项目 | 内容 | 依据 |
|------|------|------|
| 委托人/本方当事人 | （全称） | 来源文件名 |
| 对方当事人 | （全称） | 来源文件名 |
| 案由（初判） | 如：买卖合同纠纷 | 定性理由一句话 |
| 标的额 | （金额，注明计算口径） | 来源文件名+定位 |
| 程序阶段 | 诉前/一审/二审/执行 | 来源文件名 |
| 受理法院/案号 | （如有） | 来源文件名 |
| 关键期限 | 举证期限、开庭日期等（如有） | 来源文件名 |

示例片段：

```
| 案由（初判） | 买卖合同纠纷 | 双方签有《工业品买卖合同》，争议系货款支付，见《工业品买卖合同》第1、5条 |
| 标的额 | 货款本金 486,000 元（暂计，未含违约金） | 《对账单（2025.3.31）》载明未付货款，金额待与转账凭证核对 |
```

### 4.2 模块二：事实时间轴

每条格式固定为三段式，按时间升序排列：

```
日期 | 事件（一句话，主语明确） | 依据：文件名 + 页码/段落/条款定位
```

示例片段：

```
2024-03-12 | 甲乙双方签订《工业品买卖合同》，约定乙方供货总价 120 万元 | 依据：《工业品买卖合同》落款页、第3条
2024-06-20 | 甲方付款 50 万元 | 依据：《银行回单_20240620.pdf》
2024-09-(日期不清) | 乙方交付第二批货物（具体日期待核实） | 依据：《送货单（第二批）》签收栏日期模糊，建议调原件
```

时间轴纪律：日期不全的写到能确定的精度（年/年月），并标「待核实」；同日多事件按逻辑先后排列；仅有单方陈述支持的节点，依据栏写明"仅有委托人陈述，无书证"。

### 4.3 模块三：争议焦点初判

先定性，后列焦点：

```
（一）法律关系定性
本案系……法律关系（定性依据：……，见《文件》第X条）。

（二）争议焦点
焦点1：……
  - 我方可能立场：……
  - 对方可能立场：……
  - 关键证据指向：证据矩阵第X、Y项
焦点2：……
```

焦点提炼方法：从"请求权基础"反推——对方要什么钱/什么行为，支持该请求需要哪些要件事实，哪些要件事实双方有分歧，分歧点即焦点。每个焦点必须能对应到证据矩阵中的具体证据。

### 4.4 模块四：证据矩阵

| 序号 | 证据名称 | 来源文件 | 能证明什么 | 弱点/瑕疵 |
|------|----------|----------|------------|-----------|
| 1 | 《工业品买卖合同》 | 合同/合同扫描件.pdf | 买卖关系成立、货款金额及付款期限 | 仅有扫描件，原件持有情况待确认 |
| 2 | 微信聊天记录（2024.10.8） | 沟通记录/微信导出.pdf 第12页 | 对方承认收到第二批货物 | 电子数据，需保留原始载体以备质证 |

矩阵纪律：「能证明什么」必须是具体事实，不写"证明案件事实"这类空话；「弱点/瑕疵」按四类排查——形式瑕疵（复印件/无签章/无原始载体）、内容瑕疵（关键信息缺失或模糊）、关联性弱（与待证事实距离远）、来源瑕疵（取得方式可能受质疑）。无瑕疵的写"暂未发现明显瑕疵"。

### 4.5 模块五：矛盾点与疑点清单

```
矛盾点1：合同总价与对账单金额不一致
  - 《工业品买卖合同》第3条载明总价 120 万元
  - 《对账单（2025.3.31）》合计栏为 118.6 万元
  - 潜在影响：可能存在退货/折让/口头变更，直接影响诉请本金数额，必须在起草起诉状前向委托人核实
疑点1：第二批货物签收人"王某"身份不明
  - 《送货单（第二批）》签收栏为"王某"，但材料中无王某与对方公司的关系证明
  - 潜在影响：对方可能抗辩签收人无权代收，需补充王某系对方员工的证据
```

### 4.6 模块六：缺料清单

| 缺失材料 | 对应待证事实 | 建议调取方式 |
|----------|--------------|--------------|
| 对方公司工商登记信息（现状版） | 被告主体信息、法定代表人、住所地 | 国家企业信用信息公示系统打印 |
| 2024年9月后的催款记录 | 诉讼时效中断、对方违约持续状态 | 向委托人调取微信/邮件/快递底单 |

## 五、按案由的常见举证要求（缺料检查对照表）

| 案由 | 核心待证事实 | 常见必备材料 |
|------|--------------|--------------|
| 民间借贷纠纷 | 借贷合意+款项交付 | 借条/借款合同、转账凭证（大额现金交付需补强证据）、催款记录、利息约定依据 |
| 买卖合同纠纷 | 合同成立+交付+欠款金额 | 合同/订单、送货单及签收、对账单、发票、付款凭证 |
| 建设工程施工合同纠纷 | 合同关系+工程量+结算依据 | 施工合同、中标通知、签证单、验收记录、结算文件、付款凭证、（如涉实际施工人）挂靠/转包证据 |
| 服务合同纠纷 | 合同关系+服务完成+费用标准 | 服务合同、成果交付记录、验收确认、结算与付款凭证 |
| 房屋租赁合同纠纷 | 租赁关系+交付+欠租/损坏 | 租赁合同、交接清单、租金支付记录、催缴通知、现场照片 |
| 劳动争议 | 劳动关系+工资标准+争议事实 | 劳动合同、工资流水、考勤记录、社保记录、解除通知、仲裁前置文书 |
| 股权转让纠纷 | 转让合意+价款+变更登记 | 股权转让协议、股东会决议、付款凭证、工商变更记录、公司章程 |
| 侵权（财产损害） | 侵权行为+损失+因果关系 | 现场证据、报警/处理记录、损失评估或鉴定、维修票据 |

> 本表在每次梳理新类型案件后持续扩充（参照本 Skill §八 的自我更新要求）。

## 六、输出格式要求

- **主产出**：Markdown 工作底稿一份，落盘至 `案件目录/文书/卷宗梳理工作底稿_【案件简称】_YYYYMMDD.md`
- **可选产出**：用户要求时，用 python-docx 转存 .docx（A4，页边距上下 2.54cm、左右 3.18cm；一级标题黑体三号加粗，正文宋体 12pt；表格套用网格边框）
- 底稿开头固定放置声明块：

```
> 本底稿系根据案件目录内现有材料整理，仅供办案律师内部使用。
> 标注「待核实」事项须在起草正式文书前逐项核实。
> 本底稿不构成法律意见。
```

- 法条引用规范：仅写法律名称+条款号，并附提示「以现行有效版本为准，引用前核对」，不在底稿中抄录条文全文

## 七、操作 Checklist

**清点阶段：**
- [ ] 已列出案件目录全部文件清单（含子目录）
- [ ] 每个文件标注了"已读/无法读取/疑似缺页"
- [ ] 「文书/」子目录已确认存在或已创建

**通读阶段：**
- [ ] 第一轮诉讼文书已读，程序阶段已确认
- [ ] 第二轮合同及交易文件已读，法律关系已初判
- [ ] 第三轮沟通记录已读，自认/不利陈述已摘录
- [ ] 第四轮其他材料已读，主体信息已补全

**底稿质量：**
- [ ] 六大模块齐全（基本信息表/时间轴/争议焦点/证据矩阵/矛盾点/缺料清单）
- [ ] 时间轴每条均有"文件名+定位"
- [ ] 无任何未见于材料的事实表述
- [ ] 不确定事项均已标「待核实」并附核实建议
- [ ] 矛盾点均注明涉及的具体文件和潜在影响
- [ ] 缺料清单与案由举证要求逐项对照过

**输出阶段：**
- [ ] 底稿已落盘至「文书/」子目录，命名规范
- [ ] 原始材料目录未被修改
- [ ] 开头声明块已放置

## 八、注意事项

1. **底稿是活文档**：后续补充材料进入案件目录后，应增量更新底稿（时间轴插入新节点、矛盾点关闭或新增），并在底稿末尾记录更新日志（日期+更新内容）。
2. **不要替委托人做事实决定**：矛盾点的"正确版本"由委托人确认，底稿只负责暴露矛盾，不负责裁断。
3. **沟通记录引用要克制且精确**：摘录原文不超过证明目的所需范围，注明聊天双方、日期和页码；大段粘贴聊天记录是底稿质量低下的标志。
4. **敏感信息处理**：底稿中出现的身份证号、银行账号等，保留到可识别程度即可（如尾号四位），完整信息以"见原件"指引。
5. **时效意识**：通读中发现最后一次催款/还款/对账距今接近三年的，必须在矛盾点与疑点清单中以显著方式提示诉讼时效风险（《民法典》第188条，以现行有效版本为准，引用前核对）。
6. **与下游 Skill 的衔接**：本底稿的证据矩阵可直接转化为「证据清单」Skill 的输入；基本信息表与时间轴是「民事起诉状」「民事答辩状」Skill 的必读输入。起草文书前底稿中的「待核实」事项应清零或明确降级为"已向委托人确认无法补强"。
7. **语体要求**：正式书面语，禁用 emoji，禁用口语化表述；底稿虽是内部文件，仍按可被同行复核的标准撰写。
