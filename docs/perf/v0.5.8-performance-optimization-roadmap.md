## 1. 总览 / Executive Summary

当前性能优化不能只做局部微调，优先级应按用户可感知路径排序：

1. 先补齐 `v0.5.8` 基线与 release-grade evidence，避免在 `v0.5.6` fixture baseline 上误判收益。
2. 先压冷启动与首屏主包：`App` 主 chunk、全量 CSS、双语 i18n、AppShell 静态依赖是最直接瓶颈。
3. 再压实时消息路径：已有 batching、virtualization、diagnostics，但仍缺 ingress -> batch flush -> render-visible -> terminal settlement 的端到端关联证据。
4. 再压全局搜索与文件预览：当前搜索会在 query 期间重复扫描多个 provider，文件预览把 CodeMirror/PDF/语言扩展拉得过早。
5. 后端重点处理扫描、JSONL 解析、git revwalk/diff、project map walk 等重 IO/CPU 路径，保证不会阻塞 Tauri runtime 或放大前端 jank。

本文件是下一版本性能优化 backlog，按 `P0 / P1 / P2` 排序，并给出代码现状、风险、优化动作、验收标准与验证命令。

### P0-04 并行化 Bootstrap 阶段 / Parallelize and Stage Bootstrap Work

Code:

- `src/bootstrapApp.tsx`

- `src/features/composer/hooks/useInputHistoryStore.ts`

Current state:

Startup currently serializes this path:

```text
preloadClientStores
-> migrateLocalStorageToFileStore
-> initInputHistoryStore
-> import("./i18n")
-> import("./App")
-> render
-> markRendererReady
```

Risk:

- Any slow storage, migration, input history restore, or i18n load delays first render.
- The app renders only after all pre-mount work finishes.

Optimization actions:

1. Separate must-block-render work from can-run-after-render work.
2. Start `import("./App")` and i18n resource load earlier in parallel where safe.
3. Move input history restore after shell mount if composer can render with empty history and hydrate later.
4. Run localStorage migration as best-effort background task unless it is required for initial state correctness.
5. Preserve `ErrorBoundary` and bootstrap fallback behavior.
6. Extend `recordStartupMilestone` with granular milestones:
   - `bootstrap:start`
   - `storage:preload:start/end`
   - `migration:start/end`
   - `input-history:start/end`
   - `i18n:start/end`
   - `app-import:start/end`
   - `root-render:start/end`
   - `shell-ready`
7. Emit startup timing to diagnostics in development and performance builds.

Acceptance criteria:

- First render no longer waits on non-critical input history restore.
- Startup trace proves which tasks are parallelized.
- Startup failure fallback still works for critical failures.

### P0-11 File Editor 输入延迟固化 / File Editor Typing Latency Hardening

Code:

- `src/features/files/components/FileViewPanel.tsx`
- `src/features/files/components/FileViewBody.tsx`
- file editor / CodeMirror integration components under `src/features/files/components/**`
- `src/features/files/hooks/useFileExternalSync.ts`
- `src/features/workspaces/hooks/useOpenPaths.ts`
- `src/services/clientStorage.ts`
- `src/services/tauri.ts`
- `src-tauri/src/workspaces/files.rs`

Current state:

- The roadmap already covers file preview lazy loading, file tree/listing, backend scan cache, Tauri payload budget, and client storage write pressure.
- It does not yet define a dedicated P0 gate for the editor hot path after a file is open: keystroke -> CodeMirror transaction -> local dirty state -> line/selection publication -> optional save/sync -> watcher feedback.
- Existing specs cover stable file rendering, external sync guards, and delayed line-range publication, but typing latency still lacks a release budget and evidence artifact.

Optimization actions:

1. Keep CodeMirror document transactions local-first; a keystroke MUST NOT force full `FileViewPanel`, app-shell, Composer, or workspace tree recomputation.
2. Publish cursor/selection/line-range state through delayed/coalesced paths; intermediate ranges MAY be dropped if the latest range is available before send/context injection.
3. Ensure dirty buffer state stays in memory-first state; transient editor state MUST NOT be persisted through `clientStorage` or Tauri invoke on every keystroke.
4. Debounce/coalesce save, autosave, metadata, and preference writes; explicit save remains immediate, but repeated writes MUST be bounded.
5. Make `useFileExternalSync` and watcher events editor-aware: dirty buffers MUST NOT be overwritten, and self-save watcher events MUST NOT trigger immediate full reload/reparse.
6. Add content-safe typing latency diagnostics: input cadence, editor transaction duration, React commit cost where available, long-task count, IPC/write count, stale sync drops, and save debounce stats.
7. Add focused regression tests for editor typing, large-file edit mode, external-change conflict, save self-event suppression, and line-range publication coalescing.

Initial budget proposal:

| Metric | Target | Hard Fail Threshold |
|---|---:|---:|
| Editor keystroke visible echo P95 | <= 16 ms | > 32 ms |
| React commits touching file panel per keystroke | local editor subtree only | app-shell/full panel commit per key |
| Tauri invoke / FS write per keystroke | 0 | > 0 outside explicit/debounced save |
| `clientStorage` write per keystroke | 0 | > 0 for transient editor state |
| Self-save watcher reload count | 0 full reloads | any dirty buffer overwrite |
| Large-file edit P95 visible echo | <= 32 ms | > 50 ms |

Acceptance criteria:

- Typing in an already-open text file remains responsive while search hydration, file tree refresh, terminal logs, or external watcher events are active.
- Keystrokes do not trigger per-key Tauri file reads/writes, persistent storage writes, or full workspace/file-tree recomputation.
- Dirty buffers are never replaced by external sync or self-save watcher feedback without explicit user resolution.
- Performance evidence can classify file editor typing as `measured`, `proxy`, `manual-only`, or `unsupported`; proxy evidence cannot be used as release-grade proof.

## 11. 风险与 Guardrails / Risks and Guardrails

### Risk: Lazy loading can create first-open latency

Mitigation:

- Use idle preload for likely next features.
- Show feature-level skeletons.
- Measure first-open latency separately from cold start.

### Risk: CSS splitting can cause FOUC or missing styles

Mitigation:

- Define critical CSS explicitly.
- Add manual smoke matrix.
- Keep shell layout styles in bootstrap.

### Risk: i18n dynamic loading can create missing translations during mount

Mitigation:

- Block only on current locale readiness.
- Keep fallback deterministic.
- Test language switching.

### Risk: Search index can become stale

Mitigation:

- Version indexes by workspace source revisions.
- Add explicit invalidation on file/thread/message updates.
- Prefer stale-but-fast results only if UI indicates hydration state.

### Risk: Backend cache can hide external changes

Mitigation:

- Include mtime/git commit/provider home signature in cache keys.
- Provide forced refresh path.
- Log cache hit/miss and age.

### Risk: Realtime diagnostics can add overhead

Mitigation:

- Keep detailed trace in dev/perf mode.
- In production, emit bounded summary only.
- Sample or cap trace buffers.

### Risk: Event batching can hide important logs

Mitigation:

- Batch rendering, not raw capture.
- Keep export/debug raw stream available.
- Report coalesced count and truncation state clearly.

### Risk: Payload pagination can break existing UI assumptions

Mitigation:

- Introduce summary-first DTOs with explicit `hasMore`, `nextCursor`, `truncated` fields.
- Keep strict frontend mapping in `src/services/tauri.ts`.
- Add regression tests for partial payload rendering.

### Risk: Polling gates can delay important status updates

Mitigation:

- Separate critical status from decorative/background refresh.
- Always allow user-triggered manual refresh.
- Flush critical updates on focus regain and feature activation.

## 12. 不要盲目优化 / Do Not Optimize Blindly

Avoid these traps:

- Do not claim startup improvement using only fixture baseline while first paint remains unsupported.
- Do not split chunks without checking whether they are still eagerly imported.
- Do not remove batching/virtualization in realtime path to simplify code.
- Do not push all expensive work into `Promise.all`; bounded concurrency is required.
- Do not cache backend scans without invalidation keys.
- Do not treat `@ts-nocheck` AppShell splitting as cosmetic; it is a performance and correctness boundary issue.
- Do not batch terminal/runtime output without preserving raw log export.
- Do not introduce pagination without explicit loading and truncation states.
- Do not leave background polling active for hidden panels.
- Do not fix editor typing lag by disabling dirty-buffer conflict protection or external-change visibility.
